## Article 7 — Connecting an MCP Server to Copilot Studio: A Configuration Walkthrough

**Post 7 of 14**

The previous posts described what MCP and A2A enable in principle. This post walks through the actual configuration, because the gap between "supported" and "working in production" is where most makers lose time.

Microsoft Copilot Studio offers two paths to connect an MCP server. Both are documented on Microsoft Learn. The native onboarding wizard — introduced in 2025 — is now the recommended path for most cases. The alternative, a Power Apps custom connector built around the MCP operation, is still valid when you need finer control over the HTTP surface.

## Path 1 — The MCP onboarding wizard

This is the path most enterprise teams should use. It requires no Power Apps configuration and is handled entirely inside the Copilot Studio maker experience.

**Step 1 — Open the Tools page of your agent.** Every agent has a Tools tab alongside Knowledge, Topics, and Instructions.

**Step 2 — Select "Add a tool" → "New tool" → "Model Context Protocol."** This opens the MCP onboarding wizard.

**Step 3 — Provide the server details.**

- *Server name* — a short name; this is what makers see when they browse tools.
- *Server description* — this is important. The description is what the orchestrator uses at runtime to decide whether to call this server. Be specific about what the server is for and what kinds of requests it handles. A weak description will cause the orchestrator to either ignore the server or call it on requests it cannot handle.
- *Server URL* — the base HTTPS endpoint of the MCP server. Streamable HTTP transport is the supported and current transport; SSE was deprecated in August 2025.

**Step 4 — Configure authentication.** Three options are offered:

- *None* — for read-only servers on trusted networks only. Not recommended for anything touching business data.
- *API key* — the key goes in either a request header (common choice: `Authorization` with a bearer token, or `X-API-Key`) or as a query parameter. Each agent-server connection stores its own key, encrypted in the Copilot Studio connection manager.
- *OAuth 2.0* — preferred for anything involving identity. Three sub-options: dynamic discovery (simplest; works when the server supports OAuth 2.0 dynamic client registration with discovery), dynamic (for DCR without discovery), and manual (you supply authorization and token endpoints explicitly).

**Step 5 — Create the connection.** The wizard validates the server, pulls the tool catalog, and registers the tools as available to the agent. Each tool appears in the agent's tool list with its declared schema. The orchestrator can now call them.

That is the entire configuration path for the common case. The effort goes into writing good tool descriptions on the server side, not into configuring the client.

## Path 2 — Custom connector via Power Apps

Path 1 was not always available; before the native MCP integration, makers built a Power Apps custom connector with a specific `x-ms-agentic-protocol: mcp-streamable-1.0` annotation. This path still works and is the right choice when you need:

- Dynamic value lookups that feed MCP parameters from another API.
- Custom URL rewriting (for example, to route one connector to multiple regional MCP endpoints based on a catalog service).
- Fine-grained policy controls that are expressed at the Power Platform connector level, such as per-environment availability.

A public sample published by Microsoft in March 2026, "Dynamic MCP Routing in Copilot Studio," uses exactly this approach: a single connector that resolves an instance dropdown via a catalog service and rewrites the URL at runtime. It is a useful reference for anyone with multi-tenant or multi-region requirements.

Unless one of those requirements applies, Path 1 is simpler and has fewer moving parts.

## Validating the connection

The fastest way to confirm the setup is working is to ask the agent to list its tools in a test conversation, or to inspect the Tools tab. The MCP server's tools should appear by name, with their declared input schemas visible. If they do not, common causes are:

- The server URL does not end at the MCP endpoint root.
- The server does not implement the Streamable HTTP transport.
- Authentication failed silently; check the connection manager for errors.
- The server's tool manifest is malformed — the wizard validates against the MCP schema.

Once the tools appear, the remaining validation is behavioral: ask the agent to perform a task that should use the MCP tool and confirm it does.

## Tool descriptions are the production variable

One thing deserves emphasis, because it determines success in production more than any configuration detail: the description of each MCP tool, authored on the server.

A tool description in MCP should state, clearly and concisely:

- What the tool does.
- When the agent should use it (which kinds of requests).
- What it returns.
- Any side effects (writes data, sends messages, charges money).

A description like "Search items" is nearly useless to the orchestrator. "Searches the internal policy library for HR, IT, and compliance policies; returns matching passages with citations; read-only" is specific enough that the orchestrator will route appropriately.

When orchestration quality feels off — the agent is calling the wrong tool, or no tool at all — the first place to look is usually the descriptions.

## Governance at the tenant level

MCP connections are visible in the Power Platform Admin Center's Copilot Hub and in the Agent Registry introduced at Ignite 2025. Administrators can see which agents use which MCP servers, which connections use which credentials, and who owns them. DLP policies apply to MCP-backed connectors in the same way they apply to any other connector; servers that write to external systems are typically placed in a restricted business-data group.

Purview audit logs capture MCP tool invocations as part of standard Copilot Studio audit records. For regulated environments, this is usually what auditors want to see.

## A realistic example

A published walkthrough by Can Dedeoglu (February 2026) documents connecting an API-key-protected MCP server to Copilot Studio and testing across multiple models (GPT-4.1, GPT-5, Claude Sonnet) to compare how each handled the same tool-calling queries. The takeaway is that the connection setup is identical regardless of the model; the differences were in how often each model chose to call the tool, not in how the tool was wired up.

## What to do next

If you already have a backend with a REST API, the fastest route is:

1. Wrap the API in a minimal MCP server (the official MCP SDK provides starters in several languages).
2. Deploy it with HTTPS and OAuth 2.0 authentication.
3. Register it in Copilot Studio via the wizard.
4. Write crisp tool descriptions.
5. Add the tools to an agent and test.
